Architectural Decision Records

Every technology choice is documented. Click any ADR to expand the full reasoning.

Context

The frontend needs to deliver a UI for managing todos. The app is small, will not grow complex, and the primary goals are speed and simplicity.

Decision

Use plain HTML and Vanilla JavaScript with no frontend framework.

Reasons

  • →The browser already speaks HTML and JavaScript natively. No framework runtime sitting between the code and the browser.
  • →Frameworks like React ship a runtime to the browser — extra kilobytes to download, parse, and execute. Vanilla JS eliminates that entirely.
  • →AI generates vanilla JS accurately — no framework-specific patterns, version conflicts, or abstraction layers to get wrong.
  • →Instant load, zero dependencies, nothing holding it back.

Alternatives rejected

React

Ships a runtime and requires a build pipeline. Unnecessary overhead for a small app.

Next.js

Full-stack React framework. Solves problems this app does not have.

Vue 3

Same category as React. Reactive component model is unnecessary overhead here.

Tradeoff

Vanilla JS can become disorganised without discipline as the app grows. No enforced component structure. Acceptable at this scale.

Context

The app needs a styling approach. The goal is to move fast, keep the codebase simple, and avoid context switching between files.

Decision

Use Tailwind CSS utility classes for all styling.

Reasons

  • →Tailwind keeps styling in the same file as the HTML. No separate CSS files, no switching context, no inventing class names.
  • →Utility classes do exactly what they say — rounded-lg, flex, gap-4. No cascade to reason about.
  • →AI generates Tailwind accurately and consistently — no custom class names to maintain across files.

Alternatives rejected

Plain CSS

Separate files, context switching, custom class names AI generates inconsistently.

CSS Modules

Solves a scoping problem this app won't have. Added build config not worth it.

Bootstrap

Opinionated component classes. Harder to customise, heavier, less suited to a unique UI.

Tradeoff

Class names can make HTML verbose. A single element might have many classes. Readable and manageable at this scale.

Context

The app needs a backend API for todo CRUD and user auth. The question is which framework and language to use.

Decision

Use FastAPI with Python.

Reasons

  • →Auto-generates a live interactive /docs page the moment routes are written. Every endpoint testable in the browser.
  • →Pydantic validation is built in — define the data shape once, FastAPI rejects bad requests before your code runs.
  • →Async by default — faster under load, modern Python.
  • →AI writes FastAPI + SQLAlchemy + Postgres accurately.

Alternatives rejected

Flask

Blank slate — just routing. Validation and docs require extra libraries. Older choice.

Node.js + Express

Second language on the backend. No built-in validation or docs.

Next.js API Routes

Collapses frontend + backend. Couples concerns and forces React, already rejected.

Tradeoff

Python on the backend means a separate runtime from the static frontend — solved by serving the frontend from FastAPI in one Railway service.

Context

The app needs persistent storage. Todo data is structured and relational — predictable fields and user ownership.

Decision

Use PostgreSQL via Railway's native database plugin.

Reasons

  • →Railway native plugin — one click adds it, DATABASE_URL injected automatically.
  • →Todo data is structured and predictable. Exactly what a relational database is designed for.
  • →Enforces strict data integrity by default — rejects bad data rather than silently accepting it.
  • →Industry standard. SQLAlchemy pairs with it cleanly.

Alternatives rejected

SQLite

File on disk — wiped on every Railway redeploy. Not designed for concurrent writes.

MySQL

Same category but Postgres is stricter on integrity and better supported in Python ecosystem.

MongoDB

Document DB for schema-less data. Todo data is relational — a fixed schema is a feature here.

Tradeoff

Schema migrations require thought — adding a column requires a migration via Alembic. Standard practice, accepted.

Context

The app needs to be deployed somewhere accessible. The question is which hosting platform to use.

Decision

Use Railway for one app service (FastAPI serves API + frontend) plus PostgreSQL.

Reasons

  • →Already paying for it. No cost justification, no new account, no free tier limitations.
  • →Connect a GitHub repo, it detects what's running, deploys it. Push to main, it redeploys automatically.
  • →Native Postgres plugin — one click, connection string injected automatically.
  • →Entire stack lives in one Railway project.

Alternatives rejected

Vercel

Optimised for Next.js. Python backend on Vercel needs serverless config with cold start issues.

Render

Valid alternative but requires a new account with no additional benefit over an existing Railway sub.

AWS / GCP / Azure

More power but IAM, networking, and operational overhead is the wrong complexity for this app.

Tradeoff

Railway is not the cheapest if cost were the primary concern. Existing subscription removes that concern entirely.

API Surface

Endpoints no /api prefix
Auth

Todos — requires auth

POST /todos Create a new todo

PATCH /todos/{id}/uncomplete Undo complete — checkbox toggles both ways

User Stories

#1 Register

As a user, I want to register an account so that my todos are private to me.

#2 Log in

As a user, I want to log in so that I can access my todos.

#3 Log out

As a user, I want to log out so that my account is secure when I'm done.

#4 Create todo

As a user, I want to create a todo with a title, description, and priority so I can organise my tasks.

#5 View todos

As a user, I want to view all my todos so that I can see everything I need to do.

#6 Complete

As a user, I want to mark a todo as complete so I can track my progress.

#7 Edit

As a user, I want to edit a todo so I can update it if things change.

#8 Delete

As a user, I want to delete a todo so I can remove tasks I no longer need.

#9 Filter

As a user, I want to filter by priority so I can focus on what matters most.

06

Easter eggs & fun

Frontend-only polish — nothing here touches the API or database.

Hourly letter

Each hour highlights a different letter and color. The nav shows a hint; saving a todo flashes matching letters in the title for ~5 seconds.

Food words

Titles with food words (e.g. buy mango) get a matching emoji on the card and a mini emoji parade on save. PDF export skips the extra emoji when the title has words.

Action verbs

Titles with verbs like run, buy, or call make the card jump briefly on save.

Keyword easter eggs

Saving a todo whose title contains certain words triggers a floating emoji parade — one keyword per save:

walk · dog · gym · water · coffee · pizza · sleep · read · cook · email · clean · grass · run · music · shop

Chaos button

Shuffles the list, tilts the page, floats cards, spins the progress bar hue, and fires confetti. Undo restores the original order.

Completion celebrations

Sparkles on the checkbox when you check something off. Confetti when every todo is done.

Themes

Six color themes including dark mode. PDF export uses your active theme color for the header and accents.

Stats & PDF

Progress bar shows done count and percent. A line below lists priority breakdown and added-today count. Export groups open todos by added date and completed todos by done date, with timestamps on each row.

07

Reflection

What worked, what stuck, and what I'd change next time.

What went well

Writing the ADRs beforehand really helped me stay on track — for myself and for Copilot. Railway hardly gave me any issues; deploy went smoothly. And getting Google OAuth in worked well too.

From ADRs

Documenting the decisions upfront meant I wasn't second-guessing myself mid-build. We'd already decided on the important things — I could just refer back instead of re-litigating every choice.

From specs

The spec set a contract. It made sure Copilot wouldn't go off and decide something on its own — the data model, API, and user stories were already laid out.

From Copilot

First time using it — felt a bit slow at first, but was really good once I got the hang of it. The copilot-instructions file helped a lot. Dropping in screenshots of what went wrong also worked well (probably eats a lot of tokens though).

From testing with Copilot

Pretty good overall — I'd explain in detail what wasn't working and it would usually get it. But it only really understood the codebase. It wouldn't consider that the problem might be Railway's fault, or Google's. Not the best for that kind of debugging.

What I learned the most

The programmer behind it still needs to feed in direct instructions and help majorly with instructions — that's really the only way it works out. Debugging was most of what I actually did. It kept making bugs.

What I'd do differently next time

  • →Be more detailed before starting to code. I jumped in mostly right after doing the bare minimum of research.
  • →Update the specs, ADRs, and blueprint every time something changed — I didn't keep them in sync and I should have.
